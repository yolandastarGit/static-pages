## 01 产品概述（Overview）

### 1.1 产品背景

随着海外业务持续增长，销售团队每天需要处理来自官网、邮件、WhatsApp 等多个渠道的大量客户咨询。由于沟通渠道分散、客户信息不统一、销售过程缺乏标准化管理，企业普遍存在以下问题：

- 客户消息来源分散，容易遗漏潜在商机
- 销售人员依赖人工筛选询盘，效率较低
- 同一客户沟通记录分散，缺少统一沉淀
- 客户资产无法持续积累，人员离职造成客户流失
- 销售过程缺乏标准化管理，业务进展难以追踪
- 管理层缺乏统一的数据分析和经营决策依据

### 1.2 产品定位

AI 智能 CRM 是一套面向海外销售团队的客户关系管理平台。

**产品定位关键词：**

- 线索驱动
- AI 辅助
- 客户资产

AI 作为平台能力嵌入各业务场景，负责消息识别、企业信息提取、意向总结等工作，但不作为独立业务模块存在。

### 1.3 产品目标

- 多渠道客户消息统一接入
- 自动识别有效客户商机
- 标准化销售跟进流程
- 企业客户资产持续沉淀
- 销售过程可追踪、可分析、可管理

### 1.4 MVP 范围

MVP 优先实现：**流程完整、数据完整、权限完整**。

- AI、高级 BI、审批等能力逐步扩展
- 组织架构在 MVP 阶段隐藏，不保留入口，后续版本开放
- 人工认领功能预留，不启用
- 合同审批预留，不启用
- 业务邮件多服务配置预留，MVP 仅支持单一全局邮箱服务
- WhatsApp 多账号预留，MVP 仅支持单一全局 WhatsApp 账号
- 自助找回密码、MFA、SSO、企业微信/Google/Microsoft 登录预留

### 1.5 二期规划

- 人工认领公海线索
- 合同审批流程
- 组织架构管理与多级团队
- 业务关联 AI 能力在线配置
- 多邮箱服务、多 WhatsApp 账号
- 企业微信 / Google / Microsoft / LDAP / OAuth2 / SSO 登录
- MFA 多因素认证、登录设备管理、异地登录提醒
- 自助找回密码
- 业务邮件多服务、消息模板在线增删改
- 高级 BI 与自定义分析

### 1.6 产品角色

| 角色 | 数据权限范围 | 核心职责 |
|---|---|---|
| 业务员（销售业务员） | 本人本站点客户线索数据 | 维护负责的站点，管理个人邮件及 WhatsApp、查看负责线索、跟进客户、转化客户、录入合同、查看经营数据 |
| 运营专员（销售主管） | 本人及负责多个站点的所有客户线索数据 | 管理多个负责站点，查看数据、分配线索、干预线索、配置公海规则、查看站点经营分析 |
| 协同人（区域负责人） | 所管理运营专员的站点客户数据 | 查看运营的所有数据、查看运营经营分析、管理运营业务 |
| 管理员 | 所有站点客户数据 | 管理组织/用户/角色/站点、配置业务流程/AI 规则/系统参数 |

> ⚠️ 待确认项 [Q9]：`00-登录.md` §九 与本表（基线 `需求文档.md` §1.4）对运营专员/协同人数据权限范围的描述不一致，最终口径以产品确认为准。本 Master PRD 暂以基线 §1.4 为准。

### 1.7 产品设计原则

1. **线索驱动**：整个系统以线索为销售工作的核心对象。所有销售动作（分配、跟进、转客户、合同关联）均围绕线索展开。
2. **客户沉淀**：客户是企业资产。客户资料（企业信息、联系人、历史线索、历史合同）持续沉淀，不因销售人员变化而丢失。
3. **AI 辅助**：AI 是平台能力而非业务流程。系统支持系统规则识别和 AI 智能识别两种方式，AI 服务不可用时，系统仍可依据规则正常运行。
4. **多站点管理**：每个站点拥有独立的业务流程、公海规则、AI 识别规则、系统规则和联系方式，站点间数据相互隔离。
5. **销售流程标准化**：所有销售行为按照统一流程执行：消息 → 识别 → 线索 → 跟进 → 客户 → 合同。
6. **数据可追溯**：所有业务对象建立完整关联关系，支持追溯消息来源、来源站点、负责人、跟进记录、客户来源、合同来源。
7. **MVP 优先**：MVP 优先实现流程完整、数据完整、权限完整，AI、高级 BI、审批等能力逐步扩展。

### 3.3 数据权限粒度

数据权限分为 4 档粒度，应用于线索 / 客户 / 合同三个业务模块：

| 粒度 | 含义 |
|---|---|
| 仅本人 | 仅可查看自己作为负责人的数据 |
| 本团队 | 可查看所属团队所有业务员的数据 |
| 本区域 | 可查看所属区域所有业务员的数据 |
| 全部 | 可查看全公司所有业务员的数据 |

**数据权限规则：**
- 业务模块（线索/客户/合同）均应用数据权限粒度
- 合同数据权限不再按状态细分，仅按粒度 4 档
- 线索/客户/合同模块共用同一数据权限配置
- 数据权限与角色绑定，不支持手动切换为更广范围

### 3.4 站点权限

用户仅可查看已授权站点的数据。

- 用户可被授权多个站点
- 跨站点数据相互隔离
- 同一企业不同站点分别建立客户，不跨站合并

### 3.5 操作权限

| 操作 | 说明 |
|---|---|
| 新建 | 新增业务对象 |
| 编辑 | 修改已有数据 |
| 删除 | 逻辑删除（非物理删除） |
| 分配 | 分配线索/客户负责人 |
| 转客户 | 线索转化为客户（自动 + 主管手动批量） |
| 导出 | 数据导出为 Excel/CSV |
| 配置 | 系统配置项修改 |

### 3.6 角色保护规则

- 超级管理员（系统管理员）不可停用、不可删除、不可降权
- 系统必须至少保留一个启用且具备系统管理权限的管理员用户
- MVP 不提供角色继承功能，角色权限独立配置

## 04 核心业务对象

### 4.1 对象总览

系统以线索为核心业务对象，所有业务围绕以下对象展开：

```
站点 → 消息 → AI 识别 → 线索 → 客户 → 联系人 / 合同
```

### 4.2 站点

**定义：** 企业面向客户提供产品或服务的独立销售入口（如官网 A、官网 B、独立站、品牌官网）。每个站点拥有独立的业务流程、公海规则、AI 规则和系统规则。

**生命周期：** 创建 → 启用 → 修改配置 → 停用（不允许删除）

**关键字段：**

| 字段 | 说明 |
|---|---|
| 站点名称 | 站点名称 |
| 站点编码 | 唯一标识，不可重复，不可修改 |
| 域名 | 站点官网地址 |
| 联系邮箱 | 站点联系邮箱，支持多个（分号分隔） |
| WhatsApp | 站点 WhatsApp 账号 |
| 负责人 | 该站点负责人 |
| 状态 | 启用 / 停用 |
| 创建时间 | 站点创建时间 |

**关联关系：** 站点 1:N 消息、线索、客户、联系人

### 4.3 消息

**定义：** 客户与企业之间所有沟通内容的统一载体，目前支持 Email 和 WhatsApp。所有消息永久归档。

**生命周期：** 收到 → 待识别 → 识别中 → 识别完成 → 已生成线索 / 无效

**关键字段：**

| 字段 | 说明 |
|---|---|
| 消息 ID | 消息的唯一标识 |
| 消息类型 | Email / WhatsApp |
| 发件人 | 发件人信息 |
| 收件账号 | 收件账号 |
| 收发时间 | 消息的收发时间 |
| 来源站点 | 系统匹配，失败时手动修正 |
| 线索采集状态 | 未识别 / 已识别 |
| 线索生成状态 | 已生成 / 未生成 |

**关联关系：** 消息 N:1 站点、N:1 线索、N:1 联系人

**生成规则：**
- 一个线索可关联多个消息
- 一条消息最多生成一个线索
- 消息进入系统时，根据发件人信息自动创建联系人影子对象

### 4.4 线索

**定义：** 线索是整个 CRM 的核心业务对象，由 AI 智能识别生成（关闭 AI 时由人工维护）。

**生命周期：** 生成 → 进入公海 → 已分配 → 跟进中 → 高意向（系统自动判定）→ 已成交 / 无效 / 丢失

**关键字段：**

| 字段 | 说明 |
|---|---|
| 线索编号 | 唯一编号，格式 LEAD-YYYY-XXXX |
| 来源站点 | 来源站点 |
| 来源渠道 | 官网询盘 / 自然询盘 / WhatsApp / 展会 / 客户转介绍 / 其他 |
| 来源消息 | 关联的原始消息 |
| 线索联系人 | 询盘的客户名，线索发件人 |
| 当前负责人 | 用户（业务员） |
| 当前状态 | 公海待分配 / 已分配 / 跟进中 / 高意向 / 已成交 / 无效 / 丢失 |
| 线索标签 | AI 标签 + 手动标签混合 |
| AI 企业信息 | AI 提取企业信息 |
| AI 意向总结 | AI 生成的意向总结（转客户时不同步） |
| 意向产品 | AI 识别的客户感兴趣产品类目 |
| 阶段 | 取自最新跟进记录的跟进阶段 |
| 最近跟进时间 | 最近一次跟进记录创建时间 |
| 下次跟进时间 | 取自最新跟进记录 |
| 创建时间 | 创建时间 |

**关联关系：** 线索 1:N 消息、跟进记录、合同；N:1 客户、站点

**生成规则：**
- AI 判断有效 → 创建线索
- 一条消息最多生成一个线索
- 一个线索可关联多个消息
- 同一邮件含多个采购需求 → 合并为一个线索
- 同一发件人同一需求的多封邮件 → 不合并为一条线索
- 同一发件人不同需求的多封邮件 → 分别生成多条线索
- 新建线索默认进入公海池
- 关闭 AI 功能时，邮件转线索由人工维护，归属本人

### 4.5 跟进记录

**定义：** 销售围绕线索产生的所有跟进记录，存在于线索详情页及全局跟进日志页面。

**生命周期：** 新增记录 → 持续新增（不可修改，不可删除）

**关键字段：**

| 字段 | 说明 |
|---|---|
| 跟进时间 | 系统自动记录（提交时间） |
| 跟进人 | 系统自动记录（当前用户） |
| 跟进方式 | 电话 / 邮件 / WhatsApp / 会议 / 拜访 / 备注 |
| 跟进内容 | 跟进记录详细内容 |
| 客户关注 | 多选，选项由后台配置（报价/价格、交期、资质证书、产品定制能力、产品目录册、项目案例、求代理商合作、样品费及归属、是否为工厂、支付方式/条件、咨询起订量(MOQ)等） |
| 阶段 | 跟进阶段，选项由站点业务流程配置 |
| 下次跟进时间 | 计划下次跟进时间，默认今日 + 3 天 |
| 跟进附件 | 支持图片 / PDF / Word / Excel，单个文件 ≤ 10MB |

**关联关系：** 跟进记录 N:1 线索

**特殊说明：**
- 关键沟通或状态变化必须新增跟进记录
- 系统自动操作（分配、回收、转客户、状态变更、客户转移等）均自动生成跟进记录，跟进方式为"备注"
- 跟进记录全局视图入口：线索中心 → 跟进日志
- 跟进记录单线索视图入口：线索详情页 → 跟进日志 Tab
- 两个视图数据互通

### 4.6 客户

**定义：** 客户是企业资产，线索转客户后长期保存。

**生命周期：** 线索转客户 → 完善资料 → 新增联系人 → 新增合同 → 持续经营

**状态：** 跟进中 / 已成交 / 流失

**关键字段：**

| 字段 | 说明 |
|---|---|
| 客户编号 | 唯一编号，格式 CUS-YYYY-XXXX |
| 所属站点 | 客户来源的站点，创建时指定 |
| 客户名称 | 企业/客户名称 |
| 国家/地区 | 客户所在国家 |
| 行业 | 客户所属行业 |
| 官网 | 客户企业官网 |
| 客户标签 | 仅业务员手动添加的标签（客户对象不维护 AI 标签） |
| 当前负责人 | 业务负责人 |
| 客户状态 | 跟进中 / 已成交 / 流失，由业务员手动维护 |
| 最近签约合同 | 取最近一份"已签约"或"执行中"合同 |
| 最近签约合同时间 | 最近签约合同时间 |
| 创建时间 | 客户创建时间 |

**关联关系：** 客户 1:N 联系人、合同、线索

**说明：**
- 客户允许线索转化创建，也支持销售直接新建
- 客户状态由业务员手动维护，系统不自动变更（线索转客户时初始为"跟进中"，录入合同后由业务员手动变更为"已成交"，线索标记为"流失"时由业务员手动变更为"流失"）
- 客户不得物理删除

### 4.7 联系人

**定义：** 客户企业下的联系人，仅存在于客户详情。

**产生方式：**
1. 消息进入时根据发件人信息自动创建影子对象（未归客户状态）
2. 线索转客户时自动归集至目标客户
3. 客户详情页手动新增

**关键字段：**

| 字段 | 说明 |
|---|---|
| 姓名 | 联系人姓名 |
| 职位 | 职位 |
| 邮箱 | 邮箱 |
| 电话 | 电话 |
| WhatsApp | WhatsApp |
| 联系角色 | 决策人 / 采购经理 / 执行联系人 / 关键联系人 / 其他 |
| 是否主要联系人 | 是 / 否（仅可设置一位） |
| AI 识别标记 | 是否由系统从线索自动归集（只读） |
| 归属客户 ID | 未归客户时为 null |

**关联关系：** 联系人 N:1 客户

**说明：**
- 联系人归集时基于邮箱/手机号去重合并，支持人工手动调整
- 主要联系人仅可设置一位，设置新主要联系人时原主要联系人自动取消
- 联系人支持逻辑删除

### 4.8 合同

**定义：** 客户正式成交后的销售合同。

**生命周期：** 新增（已签约）→ 执行中 → 已完成 / 已终止 / 已作废

**关键字段：**

| 字段 | 说明 |
|---|---|
| 合同编号 | 人工录入，唯一，当前站点下不可重复，提交后不可修改 |
| 合同名称 | 合同名称 |
| 所属客户 | 关联客户（必填） |
| 来源线索 | 关联线索（非必填，用于成交追溯） |
| 金额 | 合同金额，单位人民币（¥） |
| 签约日期 | 合同签约日期，默认今日 |
| 合同状态 | 已签约 / 执行中 / 已完成 / 已终止 / 已作废 |
| 负责人 | 创建时默认为当前用户 |
| 合同附件 | 支持多文件上传，常见格式 |
| 创建时间 | 创建时间 |

**关联关系：** 合同 N:1 客户、N:1 线索

**说明：**
- 新增合同必须关联客户，否则禁止保存
- 保存即"已签约"，无需审批
- 合同审批能力预留，本版本不启用
- 仅已作废合同允许物理删除，其他状态合同不得物理删除
- 已签约、执行中、已完成、已终止状态的合同保留历史数据

### 4.9 公海池

**定义：** 公海池并非独立实体，表示当前没有负责人的线索集合，本质是线索的一种归属状态。

**关键字段：** 共享线索全部属性，额外维护：

| 字段 | 说明 |
|---|---|
| 入池时间 | 线索进入公海池的时间 |
| 入池原因 | 系统采集 / 人为标记 / 超期回收 |
| 回收次数 | 累计被回收次数 |

**说明：**
- 公海池按站点分别管理，不同站点的公海线索相互隔离
- 业务员只读查看本人站点公海线索，主管/管理员可分配

### 4.10 用户

**定义：** 系统登录账号，关联角色、站点、邮箱、WhatsApp 绑定。

**关键字段：**

| 字段 | 说明 |
|---|---|
| 用户 ID | 用户唯一标识 |
| 姓名 | 用户姓名 |

## 05 核心业务流程

### 5.1 总体流程

```
账号绑定 → 消息接收 → 消息识别 → 线索创建 → 公海池 → 分配 → 跟进 → 高意向 → 转客户 → 合同录入 → 客户资产沉淀
```

### 5.2 用户登录流程

**开始** → 用户访问登录页 → 选择登录方式（账号密码 / 钉钉扫码）

**分支 A：账号密码登录**
→ 输入账号 + 密码 → 校验账号是否存在 →（不存在）返回"用户名或密码错误"
→（存在）校验密码 →（不匹配）返回"用户名或密码错误"
→（匹配）校验账号状态 →（停用）返回"当前账号已停用，请联系管理员"
→（启用）校验是否需手机验证码（首次登录 / 异地登录 / 安全策略触发）
→（需验证）校验验证码 →（错误/过期）返回对应提示
→（通过）生成 Token → 初始化权限 → 记录登录日志 → 进入工作台 **结束**

**分支 B：钉钉扫码登录**
→ 钉钉扫码 → 获取 OpenID → 校验是否绑定系统账号
→（未绑定）提示"当前钉钉账号未绑定系统用户，请联系管理员" **结束**
→（账号停用）提示"当前账号已停用，请联系管理员" **结束**
→（已绑定）生成 Token → 初始化权限 → 记录登录日志 → 进入工作台 **结束**

### 5.3 账号绑定流程

**开始** → 用户进入个人中心 → 账号绑定
- **邮箱绑定**：点击新增绑定 → 输入邮箱地址 → 校验格式 + 唯一性 → 绑定成功
- **WhatsApp 绑定**：点击绑定 → 输入 WhatsApp 账号 → 校验唯一性 → 绑定成功
**结束**

### 5.4 消息接收流程

**开始** → 客户发送消息 → 消息进入 CRM
→ 系统判断账号是否已绑定 →（未绑定）丢弃 **结束**
→（已绑定）保存原始消息（正文、附件、发件人、收件账号、收发时间）
→ 创建/关联联系人影子对象
→ 识别来源站点（优先根据收件邮箱自动识别）
→ 进入识别流程 **结束**

### 5.5 消息识别流程

**开始** → 消息进入识别流程
→ 系统判断 AI 能力总开关 + 站点 AI 规则
→（开启 AI）调用 AI 智能识别 → 输出意向总结、企业信息、企业画像、风险信息、AI 标签
→（AI 失败）自动降级为系统规则识别，业务不中断
→（关闭 AI）使用系统规则识别（关键词规则引擎）
→ 识别完成 **结束**

### 5.6 线索创建流程

**开始** → 识别完成
→ 系统判断是否有效
→（无效）标记为无效消息，不生成线索，支持人工干预转为线索 **结束**
→（有效）创建线索，写入 AI 信息（企业名称、联系人、邮箱、电话、来源站点、来源渠道、来源消息、AI 企业信息、AI 标签、AI 意向总结）
→ 线索默认进入公海池
→ 根据收件账号绑定关系自动分配负责人（自动分配模式）
→ 线索状态变为"已分配"
**结束**

### 5.7 公海分配流程

**开始** → 主管进入公海池 → 选中线索 → 点击分配
→ 选择目标负责人（当前站点下"在职"销售人员）
→ 校验目标负责人状态 →（不可用）提示"该负责人已不可用，请重新选择"
→ 校验线索是否仍在公海 →（已被分配）提示"该线索已被其他操作分配，请刷新后重试"
→（通过）线索负责人更新，状态从"公海待分配"变为"已分配"
→ 自动生成跟进记录（跟进方式：备注，内容："由 [操作人] 从公海分配至 [负责人]"）
**结束**

### 5.8 公海自动回收流程

**开始** → 系统定时任务每日执行（如每日凌晨 2:00）
→ 扫描所有启用自动回收的站点
→ 筛选线索状态为"已分配"或"跟进中"且最近跟进时间超过回收超时时长的线索
→ 排除状态：已成交、无效、丢失
→ 执行回收：负责人清空，状态变更为"公海待分配"
→ 自动生成跟进记录（跟进方式：备注，内容："系统自动回收至公海"）
**结束**

> 说明：自动回收按站点配置生效，站点级配置优先，系统级兜底。若站点未启用自动回收，则不执行。

### 5.9 线索跟进流程

**开始** → 销售进入线索详情/线索列表 → 点击录入跟进
→ 弹窗填写：跟进方式、跟进内容、客户关注、阶段、下次跟进时间、附件
→ 提交校验
→ 写入跟进记录，更新线索"最近跟进时间"
→ 判断阶段是否触发高意向标记
→（触发）线索状态变更为"高意向"
→（不触发）线索状态保持不变
**结束**

### 5.10 线索转客户流程（自动）

**触发条件：** 线索状态变更为"高意向"（由跟进阶段的"触发高意向"标记自动判定）

**开始** → 系统检测到线索状态变为"高意向"
→ 自动发起转客户流程
→ 校验当前站点内客户是否存在（匹配维度：客户名称，完全匹配为主，模糊匹配为辅；匹配范围：仅限当前线索所属站点）
- 匹配 0 个 → 创建新客户（归属线索来源站点）
- 匹配 1 个 → 关联已有客户
- 匹配多个 → 列表展示，由用户选择关联目标（自动流程中由系统按最高匹配度自动选择，结果通过通知推送，用户可后续调整）
→ 同步 AI 信息至客户（企业信息、企业画像、风险信息、AI 标签；**不同步意向总结**）
→ 归集联系人（基于邮箱/手机号去重合并）
→ 线索状态变更为"已成交"
→ 通过系统通知推送给线索负责人
**结束**

> **手动批量转客户：** 保留在线索列表 → 顶部【转客户】按钮，仅运营专员/主管可用，用于主管在特殊场景下强制触发批量转客户。

### 5.11 线索分配流程（重新分配）

**开始** → 主管/负责人进入线索列表或线索详情 → 点击分配线索
→ 选择新负责人 → 填写分配备注（必填）
→ 校验新负责人 ≠ 当前负责人 →（相同）提示"新负责人与当前负责人相同，无需分配"
→ 线索负责人更新为新负责人，状态不变
→ 自动生成跟进记录
**结束**

> 影响范围：仅变更当前线索的负责人，不影响该客户下其他历史线索，也不影响后续新线索归属规则。

### 5.12 客户转移流程

**开始** → 负责人/主管进入客户列表 → 点击客户转移
→ 选择新负责人 → 填写转移备注（必填）
→ 校验新负责人状态 + 与当前负责人不同
→ 客户负责人更新为新负责人
→ **历史线索负责人保持不变**
→ **该客户后续新生成的所有线索自动分配给新负责人**
→ 自动生成跟进记录
**结束**

> ⚠️ 待确认项 [Q5]：客户转移时历史线索负责人**不同步**，仅新生成线索归属新负责人。`013-系统管理.md` §7.2 的"客户负责人变更是否同步线索负责人"参数与本规则联动，默认值取"否"。

### 5.13 合同录入流程

**开始** → 销售进入合同中心 → 点击新增合同
→ 填写：合同编号、合同名称、客户（必填）、关联线索（选填）、金额、签约日期、合同状态（默认已签约）、附件
→ 校验合同编号当前站点唯一
→ 校验客户必填、金额 > 0
→ 保存即生效，状态为"已签约"，无需审批
→ 合同负责人默认为当前用户
→ 自动生成跟进记录
**结束**

### 5.14 邮件收发流程

**收件：**
**开始** → 邮件通过 IMAP 拉取 → 进入消息接收流程（5.4）→ 进入识别流程（5.5）→ 进入线索创建流程（5.6）**结束**

**发件（回复/转发）：**
**开始** → 销售在邮件详情点击回复/转发 → 系统内打开邮件撰写页（预填原邮件内容）
→ 编辑收件人、主题、正文、附件 → 发送 或 保存草稿
→ 自动保存：每 30 秒自动保存一次（内容有变化时）
→ 草稿可继续编辑、删除（软删除，保留 30 天后物理清理）
**结束**

> ⚠️ 待确认项 [Q7]：邮件回复/转发为**系统内撰写**，与草稿管理一致。`02-邮件.md` §6.1 中"跳转至系统默认邮件客户端"的描述不生效。

### 5.15 WhatsApp 沟通流程

**开始** → 客户发送 WhatsApp 消息 → 通过 Webhook 进入消息接收流程 → 进入识别流程 → 进入线索创建流程
→ 销售在 WhatsApp 模块选中会话 → 查看消息记录（默认最近 30 条，滚动加载更早 30 条）
→ 在消息输入区输入内容（Enter 发送，Shift+Enter 换行）
→ 支持上传图片/文件
→ 发送失败可重新发送或删除
**结束**

### 5.16 异常业务流程

| 场景 | 处理方式 |
|---|---|
| 无效消息 | 不生成线索。支持人工干预：销售可手动转为线索，归属本人 |
| 线索回收 | 按站点配置开启自动回收时，超时未跟进触发回收，重新进入公海 |
| 客户已存在（同站点） | 建立关联，不重复创建 |
| 跨站点客户 | 不合并，分别建立客户 |
| 合同终止 | 保留历史数据，不可物理删除 |
| 合同作废 | 可选择"仅标记作废"或"标记作废并物理删除" |
| AI 服务不可用 | 自动降级为系统规则识别，业务不中断 |
| 来源站点匹配失败 | 允许手动修正，未选择站点时阻止生成线索 |
| WhatsApp 账号解绑 | 历史会话永久保留只读，不再接收新消息 |

## 06 功能需求明细

> 章节篇幅较大，详见各模块功能明细表。每个功能包含：功能点、描述、业务规则、权限要求、优先级。

### 6.1 登录模块

#### 功能定位
提供统一身份认证机制，支持账号密码 + 钉钉扫码登录，登录后初始化权限并跳转至工作台。

#### 页面列表
- 登录页
- 退出登录（Header 头像下拉）

#### 页面功能

| 功能点 | 描述 | 业务规则 | 权限要求 | 优先级 |
|---|---|---|---|---|
| 账号密码登录 | 系统账号 + 密码登录 | 登录失败统一提示"用户名或密码错误"防止账号枚举 | 全部角色 | P0 |
| 手机验证码 | 首次登录/异地登录/安全策略触发时强制 | 验证码 6 位数字，有效期 5 分钟，连续错误 3 次后可重新获取 | 全部角色 | P0 |
| 钉钉扫码登录 | 钉钉二维码扫码登录 | 未绑定系统账号提示联系管理员；账号停用禁止登录 | 全部角色 | P0 |
| 记住账号 | 本地存储登录账号 | 仅记账号不记密码 | 全部角色 | P0 |
| 忘记密码 | 提示"请联系系统管理员重置密码" | 不提供自助重置 | 全部角色 | P0 |
| 退出登录 | 清除 Token / Session / 前端状态，跳转登录页 | 存在未保存草稿时建议提示 | 全部角色 | P0 |
| 登录超时 | 30 分钟无操作自动退出 | 时长可在系统参数配置 | 全部角色 | P0 |
| 多端同时登录 | MVP 允许同一账号多端同时登录 | 后续支持 SSO/设备管理 | 全部角色 | P0 |
| 密码策略 | 长度 ≥ 8 位，建议数字+大小写字母+特殊字符 | 长度/复杂度可配置 | 全部角色 | P0 |
| 登录日志 | 自动记录所有登录行为 | 字段：时间、账号、IP、方式、结果、浏览器、OS | 管理员查看 | P0 |

### 6.2 工作台模块

#### 功能定位
系统首页，各角色快速了解当前工作及业务情况。

#### 页面列表
- 工作台首页

#### 页面功能

| 功能点 | 描述 | 业务规则 | 权限要求 | 优先级 |
|---|---|---|---|---|

| 新增线索卡 | 选定时间内新生成线索总数 + 来源分布 | 数据为空显示"--" | 全部角色 | P0 |
| 待跟进卡 | 当前用户下次跟进时间 = 今日 的线索数 | 点击跳转线索列表（自动筛选今日待跟进） | 全部角色 | P0 |

## 07 数据模型与对象关系

### 7.1 实体关系图

```
站点
 ├── 消息（1:N）
 ├── 线索（1:N）
 ├── 客户（1:N）
 └── 联系人（1:N）

消息
 ├── 站点（N:1）
 ├── 线索（N:1）
 └── 联系人（N:1）

线索
 ├── 消息（1:N）
 ├── 客户（N:1）
 ├── 跟进记录（1:N）
 ├── 合同（1:N）
 └── 站点（N:1）

客户
 ├── 联系人（1:N）
 ├── 合同（1:N）
 └── 线索（1:N）

合同
 ├── 客户（N:1）
 └── 线索（N:1）

联系人
 └── 客户（N:1）

跟进记录
 └── 线索（N:1）

用户
 ├── 角色（N:N）
 ├── 站点（N:N，授权关系）
 ├── 邮箱绑定（1:N）
 └── WhatsApp 绑定（1:1）
```

### 7.2 数据归属规则

| 对象 | 归属维度 | 说明 |
|---|---|---|
| 站点 | 系统级 | 由管理员创建 |
| 消息 | 站点 + 收件账号 | 来源站点优先根据收件邮箱自动识别，失败时手动修正 |
| 线索 | 站点 + 负责人 | 创建时归属来源站点，自动或手动分配负责人 |
| 客户 | 站点 + 负责人 | 创建时指定站点，负责人可转移 |
| 联系人 | 客户 | 归属客户，未归客户时为影子对象 |
| 合同 | 客户 + 负责人 | 必须关联客户，负责人默认创建者 |
| 跟进记录 | 线索 | 仅归属线索，不可独立存在 |
| 公海池 | 站点 | 公海线索按站点隔离 |

### 7.3 关联规则

- 一个线索可关联多个消息；一条消息最多生成一个线索
- 一个客户可关联多个线索；一个线索仅关联一个客户（转客户后）
- 一个客户可关联多个合同；一个合同仅关联一个客户
- 一个合同可关联一个线索（选填，用于成交追溯）
- 一个客户可关联多个联系人；联系人去重合并基于邮箱/手机号
- 跨站点客户不合并，分别建立客户

### 7.4 数据流

```
线索
  ↓ 转客户（自动，高意向触发）
客户
  ↓ 录入合同
合同

线索转客户时同步：
  - AI 企业信息 → 客户 AI 画像
  - AI 标签 → 客户 AI 画像（不同步意向总结）
  - 联系人 → 归集至客户（去重合并）

客户转移时：
  - 客户负责人 → 新负责人
  - 历史线索负责人 → 不变
  - 后续新生成线索 → 自动归属新负责人
```

## 09 状态机设计

### 9.1 消息状态机

```
待识别 → 识别中 → 识别完成 → 已生成线索
                  ↓
                无效（支持人工干预转为线索，归属本人）
```

| 状态 | 进入条件 | 可执行操作 | 下一状态 |
|---|---|---|---|
| 待识别 | 消息接收 | 系统自动进入识别流程 | 识别中 |
| 识别中 | 系统启动识别 | AI/规则识别 | 识别完成 / 无效 |
| 识别完成 | 识别结束 | 创建线索 | 已生成线索 |
| 已生成线索 | 线索创建成功 | — | 终态 |
| 无效 | 识别为无效 | 人工干预转为线索 | 已生成线索 |

### 9.2 线索状态机

```
公海待分配 → 已分配 → 跟进中 → 高意向 → 已成交
                              ↓        ↓
                            无效      无效
                              ↓        ↓
                            丢失      丢失
```

| 状态 | 进入条件 | 可执行操作 | 下一状态 |
|---|---|---|---|
| 公海待分配 | 线索创建 / 回收 | 分配 | 已分配 |
| 已分配 | 公海分配 / 自动分配 | 录入跟进 / 重新分配 / 修改状态 / 标记无效 / 标记丢失 | 跟进中 / 无效 / 丢失 / 公海待分配（回收） |
| 跟进中 | 录入跟进且阶段未触发高意向 | 继续跟进 / 修改状态 / 标记无效 / 标记丢失 | 跟进中 / 高意向 / 无效 / 丢失 / 公海待分配（回收） |
| 高意向 | 跟进阶段触发"触发高意向"标记 | 系统自动转客户 | 已成交 |
| 已成交 | 转客户成功 | 录入跟进 / 更新阶段（状态锁定） | 终态 |
| 无效 | 标记无效 | — | 终态 |
| 丢失 | 标记丢失 | — | 终态 |

**说明：**
- 线索转客户后不可回退
- 已成交状态锁定，不可变更，但仍允许录入跟进和更新阶段
- 公海回收按站点配置开启时触发，从"已分配"/"跟进中"回到"公海待分配"

### 9.3 客户状态机

```
跟进中 → 已成交
跟进中 → 流失
```

| 状态 | 进入条件 | 可执行操作 | 下一状态 |
|---|---|---|---|
| 跟进中 | 线索转客户 / 手动新建 | 录入合同 / 编辑 / 转移 / 标记流失 | 已成交 / 流失 |
| 已成交 | 业务员手动维护（录入合同后） | 编辑 / 转移 | 终态 |
| 流失 | 业务员手动维护（线索标记为流失时） | 编辑 / 转移 | 终态 |

**说明：** 客户状态由业务员手动维护，系统不自动变更。

### 9.4 合同状态机

```
已签约 → 执行中 → 已完成
已签约 → 已终止（仅执行中可终止）
已签约 → 已作废
执行中 → 已完成
执行中 → 已终止
执行中 → 已作废
```

| 当前状态 | 允许流转至 | 触发操作 | 备注 |
|---|---|---|---|
| 已签约 | 执行中 | 编辑 → 状态 | — |
| 已签约 | 已作废 | 更多 → 作废 | 仅"已签约"/"执行中"可作废 |
| 执行中 | 已完成 | 编辑 → 状态 | 完成时校验金额 > 0 |
| 执行中 | 已终止 | 更多 → 终止 | — |
| 执行中 | 已作废 | 更多 → 作废 | — |
| 已完成 | — | — | 终态 |
| 已终止 | — | — | 终态 |
| 已作废 | — | — | 终态 |

**说明：**
- 不保留"待签约"状态，保存即"已签约"
- 编辑弹窗合同状态下拉仅展示当前状态允许流转的目标状态
- 已作废合同可选"仅标记作废"或"标记作废并物理删除（不可恢复）"
- 作废操作需输入合同编号后 4 位确认

### 9.5 站点状态机

```
启用 → 停用
停用 → 启用
```

| 状态 | 进入条件 | 可执行操作 | 下一状态 |
|---|---|---|---|
| 启用 | 创建 / 重新启用 | 编辑 / 配置 / 停用 | 停用 |
| 停用 | 停用操作（二次确认） | 编辑 / 启用 | 启用 |

**说明：** 停用后停止接收新消息，保留历史数据。站点不允许删除。

### 9.6 用户状态机

```
启用 → 停用
停用 → 启用
```

| 状态 | 进入条件 | 可执行操作 | 下一状态 |
|---|---|---|---|
| 启用 | 创建 / 重新启用 | 编辑 / 重置密码 / 停用 | 停用 |
| 停用 | 停用操作（二次确认） | 编辑 / 启用 | 启用 |

**说明：** 停用后无法登录，名下数据同步停用只读，不进行数据转移。
