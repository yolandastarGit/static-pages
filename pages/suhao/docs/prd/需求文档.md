AI 智能 CRM 需求文档 V1.1（基线）
## 一、产品概述

### 1.1 产品背景

随着海外业务持续增长，销售团队每天需要处理来自官网、邮件、WhatsApp 等多个渠道的大量客户咨询。由于沟通渠道分散、客户信息不统一、销售过程缺乏标准化管理，企业普遍存在以下问题：

- 客户消息来源分散，容易遗漏潜在商机
- 销售人员依赖人工筛选询盘，效率较低
- 同一客户沟通记录分散，缺少统一沉淀
- 客户资产无法持续积累，人员离职造成客户流失
- 销售过程缺乏标准化管理，业务进展难以追踪
- 管理层缺乏统一的数据分析和经营决策依据

### 1.2 产品定位

AI 智能 CRM 是一套面向海外销售团队的客户关系管理平台。

**产品定位：**
> - 线索驱动
> - AI 辅助
> - 客户资产

AI 作为平台能力嵌入各业务场景，负责消息识别、企业信息提取、意向总结等工作，但不作为独立业务模块存在。

### 1.3 产品目标

- 多渠道客户消息统一接入
- 自动识别有效客户商机
- 标准化销售跟进流程
- 企业客户资产持续沉淀
- 销售过程可追踪、可分析、可管理

### 1.4 产品角色

| 角色 | 核心职责 |
| :--- | :--- |
| 销售业务员 | 管理个人邮件及 WhatsApp、查看负责线索、跟进客户、转化客户、录入合同、查看经营数据 |
| 销售主管 | 查看团队数据、分配线索、干预线索、配置公海规则、查看团队经营分析 |
| 区域负责人 | 查看区域所有数据、查看区域经营分析、管理区域业务 |
| 系统管理员 | 管理组织/用户/角色/站点、配置业务流程/AI 规则/系统参数 |

### 1.5 产品设计原则

**一、线索驱动**
整个系统以线索为销售工作的核心对象。所有销售动作（分配、跟进、转客户、合同关联）均围绕线索展开。

**二、客户沉淀**
客户是企业资产。客户资料（企业信息、联系人、历史线索、历史合同）持续沉淀，不因销售人员变化而丢失。

**三、AI 辅助**
AI 是平台能力而非业务流程。系统支持系统规则识别和 AI 智能识别两种方式，AI 服务不可用时，系统仍可依据规则正常运行。

**四、多站点管理**
每个站点拥有独立的业务流程、公海规则、AI 识别规则、系统规则和联系方式，站点间数据相互隔离。

**五、销售流程标准化**
所有销售行为按照统一流程执行：消息 → 识别 → 线索 → 跟进 → 客户 → 合同。

**六、数据可追溯**
所有业务对象建立完整关联关系，支持追溯消息来源、来源站点、负责人、跟进记录、客户来源、合同来源。

**七、MVP 优先**
MVP 优先实现流程完整、数据完整、权限完整，AI、高级 BI、审批等能力逐步扩展。


## 二、核心业务对象

### 2.1 对象总览

系统以线索为核心业务对象，所有业务围绕以下对象展开：

> 站点 → 消息 →（系统规则识别 / AI 识别）→ 线索 → 客户 → 联系人 / 合同

### 2.2 站点

企业面向客户提供产品或服务的独立销售入口（如官网 A、官网 B、独立站、品牌官网）。

每个站点拥有独立的联系方式、邮箱、业务流程、公海规则、AI 规则和系统规则。

**生命周期：** 创建 → 启用 → 修改配置 → 停用（不允许删除）

**核心属性：**

| 属性 | 说明 |
| :--- | :--- |
| 站点名称 | 站点名称 |
| 站点编码 | 唯一标识 |
| 官网地址 | 站点 URL |
| 联系邮箱 | 联系邮箱 |
| WhatsApp | 联系方式 |
| 状态 | 启用 / 停用 |
| 创建时间 | 站点创建时间 |

### 2.3 消息

客户与企业之间所有沟通内容的统一载体，目前支持 Email 和 WhatsApp。

所有消息永久归档。

**生命周期：** 收到 → 识别中 → 识别完成 → 归档

**核心属性：**

| 属性 | 说明 |
| :--- | :--- |
| 消息 ID | 消息的唯一标识 |
| 消息类型 | Email / WhatsApp |
| 发件人 | 发件人信息 |
| 收件账号 | 收件账号 |
| 收发时间 | 消息的收发时间 |
| 线索采集状态 | 未识别 / 已识别 |
| 线索生成状态 | 已生成 / 未生成 |

**说明：**
- 一个线索可关联多个消息
- 一条消息最多生成一个线索
- 消息进入系统时，根据发件人信息自动创建联系人影子对象

### 2.4 线索

线索是整个 CRM 的核心业务对象，由系统规则识别或 AI 智能识别生成。

**生成规则：**
- 命中系统规则或 AI 判断有效 → 创建线索
- 一条消息最多生成一个线索
- 一个线索可关联多个消息

**特殊场景：**
- 同一邮件含多个采购需求 → 合并为一个线索
- 同一发件人同一需求的多封邮件 → 合并为一条线索
- 同一发件人不同需求的多封邮件 → 分别生成多条线索

**生命周期：** 生成 → 进入公海 → 已分配 → 跟进中 → 高意向 → 转客户 / 无效 / 丢失

**核心属性：**

| 属性 | 说明 |
| :--- | :--- |
| 线索编号 | 唯一编号 |
| 线索名称 | 默认企业名称 |
| 来源站点 | 来源站点 |
| 来源渠道 | 官网询盘 / 自然询盘 / WhatsApp / 展会 / 客户转介绍 / 其他 |
| 来源消息 | 关联的原始消息 |
| 当前负责人 | 用户 |
| 当前状态 | 公海待分配 / 已分配 / 跟进中 / 高意向 / 已转客户 / 无效 / 丢失 |
| AI 标签 | AI 自动生成标签 |
| AI 企业信息 | AI 提取企业信息 |
| 意向产品 | AI 识别的客户感兴趣产品类目 |
| 创建时间 | 创建时间 |

**关系：** 线索 → 消息（1:N）、客户（N:1）、跟进记录（1:N）、合同（1:N）、站点（N:1）

### 2.5 跟进记录

销售围绕线索产生的所有跟进记录，仅存在于线索详情页。

**生命周期：** 新增记录 → 持续新增（不可修改，不可删除）

**核心属性：**

| 属性 | 说明 |
| :--- | :--- |
| 跟进时间 | 跟进时间 |
| 跟进人 | 跟进人 |
| 跟进方式 | 电话 / 邮件 / WhatsApp / 会议 / 拜访 / 备注 |
| 跟进内容 | 跟进内容 |
| 阶段 | 已联系 / 样品阶段 / 需求确认 / 合同已成交等（后台配置） |
| 下次跟进时间 | 计划下次跟进时间 |

### 2.6 客户

客户是企业资产，线索转客户后长期保存。

**生命周期：** 线索转客户 → 完善资料 → 新增联系人 → 新增合同 → 持续经营

**核心属性：**

| 属性 | 说明 |
| :--- | :--- |
| 客户编号 | 唯一编号 |
| 企业名称 | 企业名称 |
| 国家 | 国家 |
| 行业 | 行业 |
| AI 企业信息 | AI 提取企业信息 |
| AI 标签 | AI 标签 |
| 当前负责人 | 负责人 |

**关系：** 客户 → 联系人（1:N）、合同（1:N）、线索（1:N）

### 2.7 联系人

客户企业下的联系人，仅存在于客户详情和全局搜索结果。

**产生方式：**
1. 消息进入时根据发件人信息自动创建影子对象（未归客户状态）
2. 线索转客户时自动归集至目标客户
3. 客户详情页手动新增

**核心属性：**

| 属性 | 说明 |
| :--- | :--- |
| 姓名 | 联系人姓名 |
| 职位 | 职位 |
| 邮箱 | 邮箱 |
| 电话 | 电话 |
| WhatsApp | WhatsApp |
| 是否主要联系人 | 是 / 否 |
| 归属客户 ID | 未归客户时为 null |

### 2.8 合同

客户正式成交后的销售合同。

**生命周期：** 新增 → 已签约 → 执行中 → 已完成 / 已终止 / 已作废

**核心属性：**

| 属性 | 说明 |
| :--- | :--- |
| 合同编号 | 合同编号 |
| 合同名称 | 合同名称 |
| 客户 | 关联客户 |
| 来源线索 | 来源线索 |
| 金额 | 合同金额 |
| 币种 | 币种 |
| 合同状态 | 已签约 / 执行中 / 已完成 / 已终止 / 已作废 |

**说明：** MVP 录入即生效，审批能力预留。新增合同必须同时关联客户和线索，否则禁止保存。

### 2.9 公海池

公海池并非独立实体，表示当前没有负责人的线索集合，本质是线索的一种归属状态。

**核心属性：** 共享线索全部属性，额外维护入池时间、回收原因、回收次数。

## 四、核心业务流程

### 总体流程

> 账号绑定 → 消息接收 → 消息识别（规则/AI）→ 线索创建 → 公海池 → 自动分配 → 跟进 → 高意向 → 转客户 → 合同录入 → 客户资产沉淀

### 4.1 账号绑定

系统仅处理已绑定账号的消息。

- 邮箱：支持绑定多个
- WhatsApp：MVP 支持绑定一个（一个系统账号仅绑定一个 WhatsApp，反之亦然）

### 4.2 消息接收

客户发送消息 → 消息进入 CRM → 保存原始消息 → 创建/关联联系人影子对象 → 进入识别流程

系统自动完成：
- 保存消息正文、附件、发件人信息
- 创建联系人影子对象
- 保存收件账号
- 识别来源站点（优先根据收件邮箱自动识别，失败时允许手动修正）

### 4.3 消息识别

系统支持两种识别方式，互不依赖：

**系统规则识别：**
基于后台配置的关键词规则引擎判断，包括关键词匹配、域名白名单/黑名单、附件类型识别、自定义规则组合、规则优先级排序及启用/停用。

**AI 智能识别：**
自动完成意向总结、企业信息提取、企业画像、风险信息、AI 标签。

**AI 异常处理：** AI 服务异常时自动使用规则识别，业务不中断。

### 4.4 线索创建

识别成功后自动创建线索，写入 AI 信息，进入公海池。

线索自动保存：企业名称、联系人、邮箱、电话、来源站点、来源渠道、来源消息、AI 企业信息、AI 标签、AI 意向总结。

### 4.5 线索分配

| 模式 | 说明 |
| :--- | :--- |
| 自动分配（MVP 默认） | 根据收件账号绑定关系自动确定负责人 |
| 人工认领（预留） | 本版本仅预留，不启用 |
| 主管分配 | 销售主管直接指定负责人 |

### 4.6 线索跟进

线索分配后，销售围绕线索展开跟进，所有跟进行为记录在线索详情页。

跟进方式：邮件、WhatsApp、电话、拜访、会议、备注。

跟进记录包含跟进时间、跟进人、跟进内容、阶段、下次跟进时间。历史记录不可修改，仅允许追加。

### 4.7 线索转客户

线索达到高意向（系统自动判定，基于跟进阶段，阶段可后台配置）后，销售点击【转客户】。

系统在校验当前站点内客户是否存在后，关联已有客户或创建新客户，同步 AI 信息（不含意向总结），归集联系人并去重合并。

### 4.8 合同录入

成交后销售录入合同，必须关联客户和线索，保存即生效，无需审批。

### 4.9 异常业务流程

| 场景 | 处理方式 |
| :--- | :--- |
| 无效消息 | 归档，不生成线索。支持人工干预：销售可手动转为线索，归属本人 |
| 线索回收 | 超时未跟进触发回收（需站点配置开启），重新进入公海 |
| 客户已存在（同站点） | 建立关联，不重复创建 |
| 跨站点客户 | 不合并，分别建立客户 |
| 合同终止 | 保留历史数据，不可物理删除 |


## 五、生命周期

### 5.1 消息生命周期

> 待识别 → 识别中 → 识别完成 → 已生成线索 / 无效消息 → 已归档

特殊流转：无效消息支持人工干预转为线索。

### 5.2 线索生命周期

> 公海待分配 → 已分配 → 跟进中 → 高意向（系统自动判定）→ 已转客户
     → 无效 / 丢失

**高意向判定：** 系统基于跟进阶段自动判定（如“已打样”），阶段后台可配置。

**特殊流转：** 公海回收由站点配置控制，超时未跟进重新进入公海。

**不可逆：** 线索转客户后不可回退。

### 5.3 客户生命周期

> 潜在客户 → 已成交客户

客户状态由业务员手动维护，系统不自动变更。客户不得物理删除。

### 5.4 合同生命周期

> 已签约 → 执行中 → 已完成 / 已终止 / 已作废

所有历史记录保留，不可物理删除。

### 5.5 站点生命周期

> 启用 → 停用

停用后停止接收新消息，保留历史数据。站点不允许删除。


## 六、核心业务规则

### 6.1 用户与账号绑定

| 编号 | 规则 |
| :--- | :--- |
| BR-001 | 支持系统账号+密码登录和钉钉扫码登录 |
| BR-002 | 每位销售可绑定多个邮箱，所有邮箱消息统一归属于当前用户 |
| BR-003 | MVP 每位销售仅绑定一个 WhatsApp，一个系统账号仅绑定一个 WhatsApp，反之亦然 |
| BR-004 | 系统仅接收已绑定账号的消息 |

### 6.2 站点管理规则

| 编号 | 规则 |
| :--- | :--- |
| BR-005 | 站点唯一（编码或域名） |
| BR-006 | 所有业务对象必须属于一个站点 |
| BR-007 | 不同站点数据隔离，同一企业不同站点分别建立客户，不跨站合并 |
| BR-008 | 来源站点优先根据收件邮箱自动识别，失败时允许手动修正 |
| BR-009 | 站点停用后停止接收新消息，保留历史数据 |

### 6.3 消息管理规则

| 编号 | 规则 |
| :--- | :--- |
| BR-010 | 所有消息永久保存，不得物理删除 |
| BR-011 | 消息必须记录来源站点、收件账号、发件人、时间 |
| BR-012 | 消息进入时根据发件人自动创建或关联联系人影子对象 |
| BR-013 | 系统支持规则识别和 AI 识别，两者互不依赖 |
| BR-014 | 规则引擎基于关键词，支持站点维度配置关键词、域名白/黑名单、附件类型、AND/OR 组合、优先级及启用/停用 |
| BR-015 | AI 异常时自动使用规则识别，业务不中断 |
| BR-016 | AI 输出意向总结、企业信息、企业画像、风险信息、AI 标签 |
| BR-017 | AI 识别的新信息触发自动更新；手动编辑字段优先级高于 AI，手动修改后 AI 不再覆盖 |
| BR-018 | 批量 AI 提炼仅支持同一邮箱，不可跨邮箱，单次最多 50 封 |
| BR-019 | AI 提炼结果可直接下载 PDF，不保存系统 |

### 6.4 线索管理规则

| 编号 | 规则 |
| :--- | :--- |
| BR-020 | 命中系统规则或 AI 判断有效即创建线索 |
| BR-021 | 一条消息最多生成一个线索 |
| BR-022 | 一个线索可关联多个消息 |
| BR-023 | 同一邮件多需求合并为一个线索；同一发件人同一需求合并；不同需求分别生成 |
| BR-024 | 新建线索默认进入公海池 |
| BR-025 | 线索任意时刻仅允许一个负责人 |
| BR-026 | MVP 默认按收件账号绑定关系自动分配 |
| BR-027 | 人工认领功能预留，本版本不启用 |
| BR-028 | 销售主管可重新分配线索 |
| BR-029 | 公海回收按站点配置（开关、时长、是否允许再次分配） |
| BR-030 | 每次关键沟通或状态变化必须新增跟进记录 |
| BR-031 | 跟进记录仅允许新增，不修改历史，不删除 |
| BR-032 | 线索达到高意向方可转客户 |
| BR-033 | 高意向由系统基于跟进阶段自动判定，阶段后台可配置 |
| BR-034 | 转客户时校验当前站点客户是否存在，存在则关联，不存在则创建 |
